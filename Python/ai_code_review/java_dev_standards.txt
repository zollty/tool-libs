规范一（基础要求）：
1、类名/函数名要命名清晰，比较复杂的主要的类或函数要写总体描述、作者及时间，对Controller接口参数实体类、数据库实体类、枚举值等重要属性字段要备注其中文名称，对重点关键的代码行或者片段要写清楚注释。【强制要求】
2、代码上下文逻辑正确，不存在逻辑缺陷（会造成业务BUG），需要名称与代码、代码与代码、注释与代码来综合判断，确保代码逻辑是上下代码一致的、符合函数定位的、与描述匹配的。【强制要求】


规范二（项目组重点要求）：
1、对Controller接口的重点参数要进行取值校验（包括是否为空、内容是否符合要求）。
2、异常处理规范：使用`return Result.fali(msg)` 返回可以预判的业务逻辑处理失败，或者使用`new AppExcetion(msg)`返回处理错误及异常，并对常见的错误返回场景，定义错误码（在枚举类例如ErrorMsg中定义错误码及错误描述，然后使用Result或者AppException返回，例如`Result.fail(ErrorMsg.XX_CODE)` 或`new AppExcetion(ErrorMsg.XX_CODE)` ）。
3、日志规范：在关键if-else业务处理分支或者异常发生的地方，以及其他一些重要操作或容易出错的地方，添加关键的日志记录（debug或者info）。


规范三（业界通用、公司一般性要求）：
1、参考常见的Sonar代码规范、数据库设计规范、阿里巴巴Java开发手册等，包括Java的命名问题、风格问题、性能问题、潜在缺陷等，太多了，这里不一一列举，特别指出一点就是代码复杂度不能过高，单个函数体一般不超过150行，for循环嵌套一般不超过3层。针对这些常见的规范，我们只检查重点及阻塞性问题或缺陷，忽略一般、中等、轻微性问题。
2、重点检查数据库SQL开发规范，如下：
1）【强行】禁止跨多个数据库联合查询
2）【建议】尽量不要一个SQL查询3个以上的数据表。复杂统计类的系统可以适当放开，但是需要写清楚注释。
3）【建议】尽量不要一个SQL查询嵌套2层及以上的子查询
4）【建议】尽量不使用数据库自带的非常用通用的SQL函数，尽量把逻辑实现在应用程序里。
5）【建议】禁止使用select  *，select只获取需要的字段。建议在SELECT语句中只获取实际需要的字段；
6）【建议】避免单条大SQL，应在写法等价的情况下拆解成多条小SQL，以充分利用多核CPU的处理能力；
7）【建议】禁止在mysql中使用几个大表进行JOIN，且JOIN的时候又没有where过滤掉大量数据。
